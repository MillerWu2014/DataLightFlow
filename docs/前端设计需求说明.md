# DataLight QA 工作台 — 前端设计需求说明

> 版本：v0.3（含 `ui/` 演示实现对照）  
> 依据：`src/datalight/` 当前实现及 `docs/技术实现方案.md`  
> 目标读者：产品、前端、后端（HTTP 封装层）  
> **前端演示**：`ui/`（Vite + React，Mock + localStorage，见 §14）

---

## 1. 背景与目标

DataLight 已实现 **Markdown → 切块 → QA 生成 →（可选）四维评估与过滤 →（可选）扩写 / Think → Alpaca 导出** 的 Python 流水线，对外入口为 `DatalightService` / `datalight.service` 模块函数。**HTTP API 尚未建设**；仓库内已有 **`ui/` 桌面演示前端**（Mock 数据 + localStorage），用于验证信息架构与 P0 交互，对接方式见 §7、§14。

本需求说明定义一套 **QA 工作台** 前端：用户上传 Markdown，选择生成模式与参数，等待流水线完成后在界面中查看、理解质量评分，并对 QA 进行编辑、优化、删除与保存，最终导出可用训练样本。

### 1.1 产品目标

| 目标 | 说明 |
|------|------|
| 降低使用门槛 | 用可视化替代直接调用 Python API / 手工编辑 JSONL |
| 质量可控 | 四维评分与阈值过滤结果一目了然，支持人工复核 |
| 可迭代优化 | 支持单条编辑、触发扩写 / 重评、批量保存与导出 |
| 任务可追溯 | **历史任务**可浏览、重新打开，继续编辑或再次导出 |
| 与后端一致 | 字段、模式名、产物路径语义与 `pipeline/models.py`、`service.py` 对齐 |

### 1.2 非目标（首版可不实现）

- PDF / 图片摄取（MinerU）的完整向导（可后续作为「文档入库」扩展）
- Depth / Width Agentic 增强的可视化编排（可列为 P2）
- 多用户权限、审计日志、版本协作
- 替代专业 IDE 的全文 Markdown 编辑
- **移动端 / 窄屏 Web**（仅桌面 ≥1280px）

## 2. 用户角色与典型场景

### 2.1 角色

| 角色 | 诉求 |
|------|------|
| 数据标注 / 领域专家 | 上传领域 Markdown，选 taxonomy 模式，按分类体系产出 QA，人工改问法与答案 |
| 算法 / 训练工程师 | 选 default / atomic，调 `min_score` 与扩写，导出 Alpaca JSONL |
| 质检人员 | 按四维分数筛选低分项，阅读 feedback，决定保留或删除 |

### 2.2 典型旅程

```text
上传 .md → 配置模式与参数 → 提交生成任务 → 查看进度
    → 浏览 QA 列表（含评分）→ 展开上下文核对 → 编辑 Q/A
    → （可选）单条扩写 / 重评 → 删除不合格项 → 保存会话 → 导出 JSONL

（回访）侧栏历史任务 → 打开某次 Job / 会话 → 继续编辑或导出
```

## 3. 与后端能力映射

前端不重复实现 LLM 逻辑，通过 **HTTP API（待建设）** 调用现有 service。下表为功能与后端 API 的对应关系（函数名见 `docs/技术实现方案.md` §4）。

| 前端能力 | 后端能力 | 备注 |
|----------|----------|------|
| 上传 Markdown | 存储文件 + `pipeline_markdown_qa` / `pipeline_markdown_multihop_qa` | 单跳写 `{output_dir}/{generator}/` |
| 模式：default / atomic / taxonomy | `generator` 参数 | taxonomy 依赖配置 `taxonomy.*` 完整 |
| 模式：多跳 | `pipeline_markdown_multihop_qa` | 无内置四维评估；评分需单独任务或 P2 |
| 语言 | `language`: `zh` / `en` / `auto` | 影响生成与 feedback 语言 |
| 切块与数量 | `chunk_words`, `overlap_words`, `question_num` | 多跳语义切块参数略有差异 |
| 质量阈值 | `min_score` | 过滤在 runner 内存完成；前端应展示「通过 / 未通过」 |
| 扩写 | `expand_qa` + `expand_mode`，或独立 `pipeline_expand_qa` | `detail` / `contextual` / `reasoning` |
| Think | `add_think` 或 `pipeline_add_think` | 导出 `think` 字段 |
| 列表数据 | 读取 `qa_scored.jsonl`（或 atomic 的 `qa_generated.jsonl`） | atomic 无四维字段 |
| 导出 | `qa_export.jsonl` 或用户编辑后的合并 JSONL | Alpaca 三字段 + metadata |

**LLM 连接**（与 CLI 文档一致）：

- 在线：`lmstudio=True` 或配置 `llm.provider` + `base_url` / `model`
- 调试：`responses_file` 静态响应（开发 / 演示环境）

前端 **不在工作台内** 持久化 LLM 密钥；默认连接与 taxonomy 等稳定配置放在 **设置页**，读取 / 修改服务端 `configs/datalight.yaml`（或等价 API）。任务级仅允许覆盖 model、timeout 等非密钥项。

## 4. 信息架构与页面结构

采用 **主工作台 + 设置页** 两页结构（桌面 Web，**不考虑移动端适配**）。最小宽度建议 **1280px**，布局按固定三栏 / 双栏设计，不做窄屏折行或底部 Tab。

| 路由 | 名称 | 职责 |
|------|------|------|
| `/` | 重定向 | → `/workspace` |
| `/workspace` | QA 工作台 | 历史任务、上传、配置、当前任务、QA 列表、编辑、导出 |
| `/settings` | 设置 | 默认 LLM、输出目录、taxonomy 只读预览、（可选）调试项 |

顶栏全局导航：**工作台** | **设置**；当前页高亮。

### 4.1 工作台布局

```text
┌──────────────────────────────────────────────────────────────────┐
│ 顶栏：DataLight · [工作台|设置] · 当前任务标题 · 未保存 · 保存 · 导出 │
├──────────────┬─────────────────────────┬─────────────────────────┤
│ 左栏（固定宽）│ 中栏（主内容）            │ 右栏（可选，详情抽屉）    │
│              │                         │                         │
│ 【历史任务】  │  · 无选中：上传 + 生成配置 │  选中 QA：              │
│  搜索/筛选   │  · 有任务：进度条 / 列表   │  详情 / 编辑 / 四维评分 │
│  任务列表    │  · 列表上方：筛选器       │  依据段落               │
│  · 状态徽章  │                         │                         │
│  · 时间/模式 │                         │                         │
│              │                         │                         │
│ 【当前文档】  │                         │                         │
│  上传/替换   │                         │                         │
└──────────────┴─────────────────────────┴─────────────────────────┘
```

- **历史任务**占左栏上半区，始终可见（可滚动）；**当前文档**与「开始生成」占左栏下半区。
- 选中历史任务后，中栏加载该任务关联的 **Session**（QA 列表）；顶栏显示任务名 / 文档名 / 状态。
- 右栏：选中 QA 后以 **固定右栏** 展开（约 400px，`--detail-w`）；无列表数据时不占位。

（`ui/` 当前实现：有 QA 列表时右栏常驻，未选中时显示占位引导。）

### 4.2 设置页布局

```text
┌─────────────────────────────────────────────────────────────┐
│ 顶栏：同工作台导航                                            │
├─────────────────────────────────────────────────────────────┤
│  LLM：provider · base_url · model · timeout（只读/可编辑项） │
│  输出：默认 output.root 说明                                  │
│  Taxonomy：是否 complete · topic · 一级分类数量（只读摘要）    │
│  [保存设置]（写回服务端配置或提示重启生效）                    │
└─────────────────────────────────────────────────────────────┘
```

设置页变更 **不影响** 已落盘的历史任务快照；仅影响 **新提交** 的 Job。

### 4.3 历史任务与当前任务的关系

| 概念 | 说明 |
|------|------|
| **Job** | 一次流水线运行（生成）；状态：`queued` / `running` / `succeeded` / `failed` / `cancelled` |
| **Session** | 可编辑的 QA 工作副本，通常由成功 Job 初始化；保存后仍绑定来源 `jobId` |
| **历史任务列表项** | 展示 Job + 关联 Session 摘要；失败 Job 也可列出，便于查看错误与重试 |

同一 Markdown 多次生成 → 历史中出现 **多条任务**（不同 `jobId`、参数快照、创建时间）。用户从列表切换任务时，若有未保存编辑，按 §8.4 拦截。

## 5. 功能需求

### 5.1 文档上传与管理

| ID | 需求 | 优先级 |
|----|------|--------|
| F-UP-01 | 支持拖拽或点击上传 **单个** `.md` 文件；限制扩展名与大小（上限由后端配置） | P0 |
| F-UP-02 | 展示文件名、大小、上传时间；支持 **替换文档**（替换后需重新生成，旧 QA 会话需确认清除） | P0 |
| F-UP-03 | 可选：Markdown 只读预览（渲染标题与段落，非必需全文编辑） | P1 |
| F-UP-04 | P2：多文件 / 目录批量（对应路线图「目录级 QA」） | P2 |

**校验**：空文件、非 UTF-8、超大文件由后端返回明确错误文案，前端 toast 展示（Geist 文案风格：发生了什么 + 下一步）。

### 5.2 生成模式与参数配置

#### 5.2.1 流水线类型（互斥单选）

| UI 标签 | 内部值 | 说明 |
|---------|--------|------|
| 单跳 · 通用 Text2QA | `singlehop` + `generator=default` | 两阶段 meta-prompt，带四维评估 |
| 单跳 · 原子任务（高验证） | `singlehop` + `generator=atomic` | 无四维评估；展示 recall / golden-doc 相关 metadata（若有） |
| 单跳 · 分类体系 Taxonomy | `singlehop` + `generator=taxonomy` | 展示 L1/L2、task_type、reasoning_style |
| 多跳推理 | `multihop` | 语义切块 + info-pair；首版列表字段含 `reasoning_steps`、`supporting_facts` |

#### 5.2.2 通用参数

| 参数 | 控件 | 默认 | 说明 |
|------|------|------|------|
| 目标语言 | 下拉 | `zh` | `zh` / `en` / `auto` |
| 每块最大词数 | 数字输入 | 单跳 512 / 多跳 800 | 与 service 默认一致 |
| 块重叠词数 | 数字输入 | 0 | 多跳语义切块下可灰显说明「未使用」 |
| 每 chunk 最多 QA 数 | 数字输入 | 1 | `question_num` |
| 最低质量分 | 滑块或数字 1–5 | 3.0 | `min_score`；atomic / 多跳首版可隐藏 |
| atomic 每块结论上限 | 数字 | 10 | 仅 atomic 显示 |

#### 5.2.3 后处理开关（单跳）

| 开关 | 参数 | 说明 |
|------|------|------|
| 生成后扩写 | `expand_qa` | 开启后展示扩写问法/答法 |
| 扩写模式 | `expand_mode` | `detail` / `contextual` / `reasoning` |
| 生成 Think | `add_think` | 展示可折叠的 `think` 与最终 output |

**提交**：「开始生成」创建异步 **Job**；运行中禁用重复提交或提供「取消」（若后端支持）。

### 5.3 任务进度与错误

| ID | 需求 | 优先级 |
|----|------|--------|
| F-JOB-01 | 展示阶段：切块 → 生成 → 评估 → 过滤 → 扩写 → Think → 导出（按实际启用阶段动态显示） | P0 |
| F-JOB-02 | 长任务显示不确定进度 + 已耗时；支持 WebSocket 或轮询 `job status` | P0 |
| F-JOB-03 | 失败时展示阶段名、错误码、可读 message；保留已完成的中间 JSONL 下载链接（若后端提供） | P1 |
| F-JOB-04 | 见 §5.8 历史任务 | P0 |

### 5.4 QA 列表展示

#### 5.4.1 列表形态

- **默认**：卡片列表，每卡片含：问题摘要（2 行截断）、综合分或四维迷你条、标签（`qa_type`、`hop_type`、taxonomy 标签）、状态（已通过阈值 / 未通过 / 已编辑 / 已删除待保存）。
- **可选视图**：紧凑表格（适合批量质检）。

#### 5.4.2 单条记录必展示字段

| 字段组 | 字段 | 展示方式 |
|--------|------|----------|
| 定位 | `source_md`, `chunk_index` | 次要 metadata；链接到上下文 |
| 核心 | `question`, `answer` | 主内容；扩写存在时并列或 Tab：`expanded_*` |
| 上下文 | `context` 或 `chunk_text` | 抽屉内折叠面板「依据段落」 |
| 类型 | `hop_type`, `qa_type` | Badge |
| Taxonomy | `level1_name`, `level2_name`, `task_type`, `reasoning_style` | 仅 taxonomy 模式 |
| 多跳 | `reasoning_steps`, `supporting_facts` | 结构化列表 |
| Think | `think`, `think_status` | 折叠；失败显示 `think_error` |
| 扩写 | `expansion_status`, `expansion_notes` | 失败时突出提示 |

#### 5.4.3 四维质量评估（非 atomic 单跳）

与 `QAScoredRecord` 一致，分数 **1–5**（可为整数或浮点），每条维度展示 **分数 + 短 feedback**：

| 维度 key | 中文名 | 含义（与 scorer 对齐） |
|----------|--------|-------------------------|
| `question_quality` | 问题质量 | 清晰度、有意义 |
| `answer_alignment` | 答案对齐 | 是否直接回答问题 |
| `answer_verifiability` | 答案可验证性 | 相对上下文是否可客观验证 |
| `downstream_value` | 下游价值 | 训练 / RAG 可用性 |

**可视化建议**：

- 四维雷达图或四段水平条（色阶：1–2 红、3  amber、4–5 绿，须同时显示数字，不单靠颜色）。
- **综合分**：四维算术平均或最低分（需在 UI 注明规则；默认与过滤逻辑一致：任一分 < `min_score` 则标记「未通过过滤」）。
- 点击维度展开完整 `*_feedback` 文本。

**atomic 模式**：隐藏四维区域，改为说明「已在生成阶段完成 recall / golden-doc 验证」；若后端返回验证分数字段，再补充展示。

**多跳首版**：若无评分 JSON，列表不显示四维；P1 增加「对选中条发起评估」调用 evaluator 子任务。

### 5.5 筛选与排序

| 能力 | 说明 |
|------|------|
| 按最低分 / 单维度分 | 如「下游价值 < 3」 |
| 按 taxonomy 标签 | L1 / task_type |
| 按 chunk_index | 顺序浏览文档 |
| 仅显示通过阈值 | 对应 runner 过滤后集合 |
| 仅显示已编辑 | 本地 dirty 标记 |
| 排序 | 综合分、chunk_index、编辑时间 |

### 5.6 编辑、优化与删除

| ID | 需求 | 优先级 |
|----|------|--------|
| F-ED-01 | 在抽屉内 **inline 编辑** `question`、`answer`；若有扩写字段，可选择编辑原问答或扩写问法（Radio：使用扩写版 / 使用原版） | P0 |
| F-ED-02 | 编辑后标记 `user_modified=true`（前端会话态，保存时写入后端） | P0 |
| F-ED-03 | **删除**：软删除（列表灰显，保存前可撤销）；硬删除在「保存」时提交 | P0 |
| F-ED-04 | **单条优化 · 扩写**：调用 `pipeline_expand_qa`（选定 mode），结果写回该条的 `expanded_*` | P1 |
| F-ED-05 | **单条优化 · 重评**：对编辑后的 Q/A 重新跑 `Text2QAEvaluatorOperator`，更新四维分数 | P1 |
| F-ED-06 | **单条优化 · Think**：对当前最终 Q/A 调用 `pipeline_add_think` | P2 |
| F-ED-07 | 批量删除、批量标记「已审核」 | P1 |

**约束**：编辑不应自动篡改 `chunk_text`；若用户修改答案与上下文严重不符，重评后 `answer_verifiability` 应下降，UI 可提示「建议核对依据段落」。

### 5.7 保存与导出

| ID | 需求 | 优先级 |
|----|------|--------|
| F-SV-01 | **保存会话**：将当前列表（含编辑、删除态）持久化为服务端 JSONL 或项目 ID；顶栏显示「未保存更改」 | P0 |
| F-SV-02 | **导出 Alpaca JSONL**：与 `AlpacaExportOperator` 字段一致（`instruction`, `input`, `output`, `source_md`, `chunk_index`, `think`, taxonomy 等） | P0 |
| F-SV-03 | 导出范围：全部 / 仅通过阈值 / 仅选中 | P0 |
| F-SV-04 | 导出前预览单条 Alpaca 样例 | P1 |
| F-SV-05 | 导出 CSV（仅 Q/A + 分数）便于 Excel 质检 | P2 |

**冲突处理**：保存失败时不丢弃本地编辑；提示重试或导出本地备份（JSON 下载）。

### 5.8 历史任务

| ID | 需求 | 优先级 |
|----|------|--------|
| F-HIS-01 | 左栏展示 **历史任务列表**（服务端持久化，刷新 / 重开浏览器后仍存在） | P0 |
| F-HIS-02 | 每条摘要：**文档文件名**、**创建时间**、**流水线**（单跳/多跳 + generator 标签）、**状态**、**QA 条数**（完成后） | P0 |
| F-HIS-03 | 点击条目 → 中栏加载对应 Session；失败任务 → 展示错误详情 +「重试」（相同 upload + 参数） | P0 |
| F-HIS-04 | 列表 **搜索**（按文件名）与 **筛选**（状态、generator、日期范围） | P1 |
| F-HIS-05 | 支持 **新建任务**（「+ 新任务」）与 **删除历史**（仅删索引与 Session，是否删磁盘 JSONL 由后端策略决定；需二次确认） | P1 |
| F-HIS-06 | 运行中任务在列表内 **实时更新** 状态与阶段（轮询或 WebSocket，与 F-JOB-02 共用） | P0 |
| F-HIS-07 | 当前选中任务在列表中高亮；顶栏标题与列表项一致 | P0 |

**排序默认**：按创建时间 **降序**（最新在上）。

**空列表**：引导上传并「开始生成」；文案示例：「尚无历史任务。上传 Markdown 并生成第一批 QA。」

## 6. 前端数据模型（会话层）

在 backend Record 之上，前端每条 QA 建议维护：

```typescript
interface QAWorkspaceItem {
  /** 稳定 id：后端 row index 或 uuid */
  id: string;
  /** 与 qa_scored.jsonl 行一致 */
  record: Record<string, unknown>;
  /** 本地状态 */
  local: {
    deleted: boolean;
    dirty: boolean;
    selected: boolean;
    filterPassed?: boolean; // 相对 min_score
  };
}

interface WorkspaceSession {
  id: string;
  sourceFileName: string;
  pipeline: "singlehop" | "multihop";
  generator?: "default" | "atomic" | "taxonomy";
  params: PipelineParamsSnapshot;
  jobId: string;
  items: QAWorkspaceItem[];
  createdAt: string;
  updatedAt: string;
}

/** 历史侧栏列表项 */
interface TaskHistoryEntry {
  jobId: string;
  sessionId: string | null;
  sourceFileName: string;
  pipeline: "singlehop" | "multihop";
  generator?: "default" | "atomic" | "taxonomy";
  status: "queued" | "running" | "succeeded" | "failed" | "cancelled";
  stage?: string;
  qaCount?: number;
  createdAt: string;
  finishedAt?: string;
  errorMessage?: string;
}
```

**Job 与 Session**：每个成功 Job 创建或绑定一个 Session；用户编辑保存只更新 Session。对同一文档再次「开始生成」会 **新增** 一条历史任务，不覆盖旧 Job（旧 Session 只读打开仍可编辑，互不影响）。

## 7. API 契约（建议，供后端实现）

当前无 HTTP 层，以下为前端依赖的 **最小 REST 形状**（路径可调整，语义需保持）。

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/uploads` | multipart `.md` → `{ uploadId, path }` |
| POST | `/api/v1/jobs/qa` | body: uploadId + pipeline 参数 → `{ jobId, sessionId? }` |
| GET | `/api/v1/jobs` | 历史任务列表，query: `q`, `status`, `generator`, `from`, `to`, `limit`, `offset` |
| GET | `/api/v1/jobs/{jobId}` | `{ status, stage, error?, resultPaths?, sessionId? }` |
| GET | `/api/v1/jobs/{jobId}/qa` | 分页 QA 列表，query: `scored=true` |
| DELETE | `/api/v1/jobs/{jobId}` | 删除历史任务记录（及关联 Session，产物策略由后端定义） |
| GET | `/api/v1/sessions/{sessionId}` | 会话元数据 + 是否有未同步的 Job 产物 |
| PATCH | `/api/v1/sessions/{sessionId}/qa/{id}` | 更新单条 |
| POST | `/api/v1/sessions/{sessionId}/qa/{id}/expand` | 单条扩写 |
| POST | `/api/v1/sessions/{sessionId}/qa/{id}/evaluate` | 单条重评 |
| DELETE | `/api/v1/sessions/{sessionId}/qa/{id}` | 删除 |
| PUT | `/api/v1/sessions/{sessionId}` | 批量保存 |
| GET | `/api/v1/sessions/{sessionId}/export` | 下载 JSONL |
| GET | `/api/v1/config` | 设置页：LLM + output + taxonomy 摘要（脱敏） |
| PUT | `/api/v1/config` | 保存可编辑配置项（设置页） |

**分页**：单文档 QA 量可能数百条，列表 API 需 `limit` / `offset` 或 cursor。

**Idempotency**：重复 `POST jobs/qa` 建议使用 `Idempotency-Key` 防双提交。

## 8. 交互与状态

### 8.1 空状态

Copy 示例（Geist Voice）：

> 尚无 QA。上传 Markdown 并选择生成模式以创建第一批问答对。

### 8.2 进行中

> 正在评估问答对…（第 12 / 48 条）

使用省略号字符 `…`，不用三点 ASCII。

### 8.3 切换历史任务（未保存）

若当前 Session 存在 `dirty` 项，切换左栏任务时：

1. 弹窗：「当前更改尚未保存。保存并切换 / 放弃更改 / 取消」  
2. 取消则保持当前任务选中  

### 8.4 删除确认

destructive 二次确认：「删除这条 QA？」按钮文案用 **Delete QA Pair**，不用 bare「确认」。

### 8.5 快捷键（P1）

- `j` / `k`：上一条 / 下一条  
- `e`：编辑  
- `Cmd+S`：保存会话

## 12. 验收标准（P0）

1. 上传合法 `.md` 并选择 `default` + `zh`，成功完成后列表条数与后端 `qa_scored.jsonl` 一致（过滤后条数与 `min_score` 一致）。
2. 每条非 atomic 记录展示四个维度分数及 feedback 摘要。
3. 修改 question 后保存，重新打开会话内容一致；导出 JSONL 中 `input`/`output` 为编辑后文本。
4. 软删除项不出现在「导出：仅通过」结果中。
5. LLM 不可用时，Job 失败且界面展示可读错误，不白屏。
6. Taxonomy 模式下列展示 `level1_name`、`task_type` 等字段。
7. atomic 模式不展示空白四维区域，且无「未评分」误导文案。
8. 完成至少两次生成后，历史列表显示两条记录；刷新页面后列表仍在，点击可打开对应 QA 列表。
9. 从 `/settings` 可返回 `/workspace`，且不在工作台内嵌全套 LLM 表单（稳定配置仅在设置页）。
