# DataLightFlow 技术实现方案

## 1. 背景与目标

DataLightFlow 是对原 DataFlow 仓库进行轻量化重构后的独立数据处理工程。当前实现聚焦两条主线：

1. 文档摄取：通过本地 MinerU 将 PDF、图片、直接 PDF URL 转换为 Markdown，并产出 `ingest_manifest.jsonl`。
2. QA 数据生成：基于 Markdown 生成、评估、过滤、扩写、补充 `think` 字段，并导出可用于训练或 RAG 的 JSONL 数据。

设计原则：

- 不依赖原 `dataflow` 运行时。
- 模块相对独立，流程可组合。
- JSONL 作为中间产物格式，便于调试、恢复和后处理。
- 默认中文 QA 生成，同时支持英文和自动跟随上下文语言。
- 增强能力默认关闭，通过独立 CLI 或可选参数启用。

## 2. 总体架构

当前核心包位于 `src/datalight/`：

```text
src/datalight/
  cli.py
  contracts/
  ingest/
  pipeline/
    core.py
    qa/
      agentic.py
      expansion.py
      filters.py
      formatters.py
      language.py
      llm.py
      multihop.py
      operators.py
      runner.py
      thinking.py
```

模块职责：

- `contracts/`：共享数据结构和错误码。
- `ingest/`：本地 MinerU 调用、目录摄取、URL 摄取、输出路径规划。
- `pipeline/core.py`：极简 `Operator` / `Pipeline` 抽象。
- `pipeline/qa/`：Markdown QA 生成、评估、过滤、导出以及可选增强。
- `cli.py`：Typer CLI 入口。

核心抽象是轻量版 Pipeline/Operator：

```python
Record = dict[str, Any]

class Operator:
    def run(self, rows: list[Record]) -> list[Record]:
        ...
```

每个 Operator 输入和输出都是 `list[dict]`，不引入复杂 storage、DAG 编译或外部框架。

## 3. 文档摄取流程

### 3.1 支持输入

当前摄取支持：

- 目录下的 PDF 和图片：`.pdf`、`.png`、`.jpg`、`.jpeg`、`.webp`、`.gif`
- 直接 PDF URL

暂不支持：

- Word：`.doc`、`.docx`
- PPT：`.ppt`、`.pptx`
- HTML URL 正文抽取

Word/PPT 在目录摄取中会标记为 `skipped`，不进入 MinerU 解析。

### 3.2 MinerU 本地调用

本地 MinerU 由 `src/datalight/ingest/mineru_local.py` 封装。

调用形式：

```bash
mineru -p <source> -o <work_root> -b <backend> --source local
```

默认 backend：

```text
vlm-auto-engine
```

MinerU 可执行文件解析规则：

1. 优先读取环境变量 `MINERU_EXECUTABLE`
2. 否则使用 `mineru`

示例：

```bash
export MINERU_EXECUTABLE=/Users/miller/miniconda3/envs/mineru/bin/mineru
```

MinerU 输出 Markdown 的解析做了兼容：

- 优先查找 `<work_root>/<stem>/<backend>/<stem>.md`
- 若不存在，回退查找 `<work_root>/<stem>/*/<stem>.md`
- 如果存在 `auto/` 输出目录，优先使用 `auto/<stem>.md`

这是为了兼容 MinerU 实际输出目录可能与传入 backend 不完全一致的情况。

### 3.3 目录摄取

CLI：

```bash
datalight ingest directory <input_dir> <output_dir> \
  --backend vlm-auto-engine \
  --timeout 3600
```

输出：

- Markdown：按输入目录结构镜像输出，扩展名改为 `.md`
- Manifest：`<output_dir>/ingest_manifest.jsonl`
- 临时目录：`<output_dir>/.datalight/mineru_work/`

默认成功后删除 MinerU 中间目录。若需要保留：

```bash
--keep-intermediate
```

### 3.4 URL 摄取

CLI：

```bash
datalight ingest url <output_dir> --url <pdf_url>
```

URL 输出路径包含 host 和 URL 指纹：

```text
urls/<host_sanitized>/<sha256_url_prefix16>/source.md
```

URL 摄取会先做 Content-Type 和 PDF 魔数检查：

- HTML URL 返回 `E_URL_HTML_NOT_SUPPORTED`
- 非 PDF 返回 `E_URL_NOT_PDF`
- 下载或 MinerU 失败返回对应错误码

### 3.5 Manifest 结构

`ingest_manifest.jsonl` 每行是一个 `IngestRecordRow`：

```json
{
  "source_path": "...",
  "output_md_path": "...",
  "status": "ok",
  "parser": "mineru_local",
  "error_code": null,
  "sha256": "...",
  "mineru_version": "...",
  "mineru_backend": "vlm-auto-engine",
  "source_kind": "file",
  "url_host": null,
  "url_fingerprint": null,
  "duration_ms": 1234,
  "stderr_tail": null,
  "error_detail": null
}
```

## 4. LLM 接口设计

LLM 统一通过 `LLMClient` 协议访问：

```python
class LLMClient(Protocol):
    def generate(self, prompts: list[str], *, system_prompt: str = "") -> list[str]:
        ...
```

已实现两个客户端：

- `StaticLLMClient`：测试和 dry-run 使用，按顺序返回预置响应。
- `OpenAICompatibleLLMClient`：使用标准库 `urllib` 调用 OpenAI-compatible Chat Completions API，适配 LM Studio。

默认 LM Studio 配置：

```text
base_url = http://127.0.0.1:1234/v1
model = gemma-4-31b-it
temperature = 0.2
```

## 5. 语言策略

语言策略由 `pipeline/qa/language.py` 统一管理。

支持：

- `zh`：强制中文，当前默认值。
- `en`：强制英文。
- `auto`：跟随上下文主语言。

CLI 参数：

```bash
--language zh
--language en
--language auto
```

中文约束会写入生成、评估、多跳、扩写和 think prompt：

```text
问题、答案、评分反馈和推理步骤必须使用中文。
```

## 6. 单跳 QA Pipeline

主入口：

```python
run_markdown_qa_pipeline(...)
```

CLI：

```bash
datalight pipeline markdown-qa \
  --markdown <file.md> \
  --output-dir <out> \
  --lmstudio \
  --language zh
```

完整默认流程：

```text
Markdown
  -> chunks.jsonl
  -> qa_generated.jsonl
  -> qa_scored.jsonl
  -> filter
  -> qa_export.jsonl
```

### 6.1 Chunk

`MarkdownChunkOperator` 读取 Markdown 文件，将文本按空白分词并切分为 chunk。

关键参数：

- `--chunk-words`：默认 `512`
- `--overlap-words`：默认 `0`

产物：`chunks.jsonl`

字段示例：

```json
{
  "source_md": "guide.md",
  "chunk_index": 0,
  "chunk_text": "..."
}
```

### 6.2 QA 生成

`Text2QAGeneratorOperator` 对每个 chunk 调 LLM 生成 QA。

关键参数：

- `--question-num`：每个 chunk 生成多少个 QA，默认 `1`
- `--language`：默认 `zh`

产物：`qa_generated.jsonl`

字段：

```json
{
  "generated_question": "...",
  "generated_answer": "..."
}
```

### 6.3 四维评估

`Text2QAEvaluatorOperator` 对 QA 执行四个维度评分：

- `question_quality`
- `answer_alignment`
- `answer_verifiability`
- `downstream_value`

每个维度都包含：

- `<dimension>_grade`
- `<dimension>_feedback`

产物：`qa_scored.jsonl`

字段示例：

```json
{
  "question_quality_grade": 5,
  "question_quality_feedback": "问题清晰",
  "answer_alignment_grade": 5,
  "answer_alignment_feedback": "答案匹配",
  "answer_verifiability_grade": 5,
  "answer_verifiability_feedback": "可验证",
  "downstream_value_grade": 5,
  "downstream_value_feedback": "有训练价值"
}
```

### 6.4 过滤

`QAFilterOperator` 支持对四个维度设置阈值。

CLI 目前使用统一阈值：

```bash
--min-score 3
```

在 `markdown-qa` 中该阈值会同时应用到：

- question quality
- answer alignment
- answer verifiability
- downstream value

### 6.5 Alpaca 导出

`AlpacaExportOperator` 输出 `qa_export.jsonl`。

导出字段：

```json
{
  "instruction": "Please answer the following question based on the provided information.",
  "input": "...",
  "output": "...",
  "source_md": "...",
  "chunk_index": 0
}
```

如果 QA 经过扩写，则导出优先使用：

- `expanded_question`
- `expanded_answer`

如果 QA 经过 think 增强，则导出额外包含：

```json
{
  "think": "..."
}
```

## 7. 多跳 QA Pipeline

主入口：

```python
run_markdown_multihop_qa_pipeline(...)
```

CLI：

```bash
datalight pipeline markdown-multihop-qa \
  --markdown <file.md> \
  --output-dir <out> \
  --lmstudio \
  --language zh
```

流程：

```text
Markdown
  -> chunks.jsonl
  -> multihop_contexts.jsonl
  -> qa_multihop_generated.jsonl
  -> qa_multihop_export.jsonl
```

### 7.1 Context 构造

`MultiHopContextBuilderOperator` 从 chunk 中切句，并构造：

- `premise`
- `intermediate`
- `conclusion`
- `related_contexts`
- `multihop_context`
- `supporting_sentence_count`

默认要求至少 3 个句子：

```bash
--min-context-sentences 3
```

### 7.2 多跳生成

`MultiHopQAGeneratorOperator` 要求 LLM 输出严格 JSON：

```json
{
  "question": "...",
  "reasoning_steps": [{"step": "..."}],
  "answer": "...",
  "supporting_facts": ["...", "..."],
  "type": "domain_tag"
}
```

### 7.3 多跳导出

`MultiHopAlpacaExportOperator` 输出：

```json
{
  "instruction": "Answer the multi-hop question using the provided supporting facts.",
  "input": "...",
  "output": "...",
  "source_md": "...",
  "chunk_index": 0,
  "metadata": {
    "reasoning_steps": [],
    "supporting_facts": [],
    "type": ""
  }
}
```

## 8. QA 扩写阶段

扩写模块位于：

```text
src/datalight/pipeline/qa/expansion.py
```

核心组件：

- `QAExpansionOperator`
- `run_qa_expansion_pipeline`
- `build_expansion_prompt`
- `parse_expansion_response`

扩写是可选阶段，不影响默认 QA 流程。

### 8.1 独立扩写

CLI：

```bash
datalight pipeline expand-qa \
  --input qa_scored.jsonl \
  --output qa_expanded.jsonl \
  --lmstudio \
  --mode detail \
  --language zh
```

支持 mode：

- `detail`：扩写答案，补充背景、边界和解释。
- `contextual`：让问题更自然、更贴近真实用户提问。
- `reasoning`：加入简短推理说明，同时保持可验证。

LLM 输出 JSON：

```json
{
  "expanded_question": "...",
  "expanded_answer": "...",
  "expansion_type": "detail",
  "expansion_notes": "..."
}
```

写回字段：

- `expanded_question`
- `expanded_answer`
- `expansion_type`
- `expansion_notes`
- `expansion_status`

失败时：

- 保留原 QA
- 写入 `expansion_status = "failed"`
- 写入 `expansion_error`

### 8.2 接入 markdown-qa

CLI：

```bash
datalight pipeline markdown-qa \
  --markdown <file.md> \
  --output-dir <out> \
  --lmstudio \
  --expand-qa \
  --expand-mode detail
```

接入位置：

```text
生成 -> 评分 -> 过滤 -> 扩写 -> 导出
```

产物：

```text
qa_expanded.jsonl
```

## 9. Think 增强阶段

Think 模块位于：

```text
src/datalight/pipeline/qa/thinking.py
```

核心组件：

- `QAThinkOperator`
- `run_qa_thinking_pipeline`
- `build_think_prompt`
- `parse_think_response`

该阶段会把相同问题输入 LLM，让 LLM 生成 `think`，并用 LLM 返回的 `answer` 重构最终答案。

### 9.1 独立运行

CLI：

```bash
datalight pipeline add-think \
  --input qa_expanded.jsonl \
  --output qa_with_think.jsonl \
  --lmstudio \
  --language zh
```

LLM 输出 JSON：

```json
{
  "think": "...",
  "answer": "..."
}
```

字段策略：

- 如果存在 `expanded_question` / `expanded_answer`，则基于扩写后的 QA 生成 think。
- 否则基于 `generated_question` / `generated_answer`。
- 如果 LLM 不输出 `think`，则写入 `think: ""`。
- 如果 LLM 输出了 `answer`，则用该答案更新最终答案。
- 原答案会保存在 `original_generated_answer` 或 `original_expanded_answer`。

失败时：

- 保留原 QA
- 写入 `think: ""`
- 写入 `think_status = "failed"`
- 写入 `think_error`

### 9.2 接入 markdown-qa

CLI：

```bash
datalight pipeline markdown-qa \
  --markdown <file.md> \
  --output-dir <out> \
  --lmstudio \
  --add-think
```

可以和扩写组合：

```bash
datalight pipeline markdown-qa \
  --markdown <file.md> \
  --output-dir <out> \
  --lmstudio \
  --expand-qa \
  --add-think
```

组合顺序：

```text
生成 -> 评分 -> 过滤 -> 扩写(可选) -> Think(可选) -> 导出
```

产物：

```text
qa_with_think.jsonl
```

## 10. 其它 QA 能力

### 10.1 QAExtractor 风格格式转换

模块：

```text
src/datalight/pipeline/qa/formatters.py
```

`QAExtractorOperator` 支持将嵌套 `qa_pairs` 结构展平为 Alpaca 风格记录。

输入示例：

```json
{
  "qa_pairs": [
    {
      "question": "...",
      "answer": "...",
      "reasoning_steps": [],
      "supporting_facts": []
    }
  ]
}
```

输出示例：

```json
{
  "instruction": "Please answer the following question based on the provided information.",
  "input": "...",
  "output": "...",
  "metadata": {
    "reasoning_steps": [],
    "supporting_facts": [],
    "type": ""
  }
}
```

### 10.2 去重与多样性过滤

模块：

```text
src/datalight/pipeline/qa/filters.py
```

`QADedupFilterOperator` 支持：

- exact duplicate 去重
- 基于词集合 Jaccard similarity 的轻量近重复过滤

默认相似度阈值：

```text
0.92
```

### 10.3 Depth / Width QA

模块：

```text
src/datalight/pipeline/qa/agentic.py
```

已实现两个轻量 Operator：

- `DepthQAGeneratorOperator`：基于单条 QA 生成更深层追问。
- `WidthQAGeneratorOperator`：基于相邻 QA 窗口生成横向关联问题。

这部分目前作为独立 Operator 提供，尚未默认接入 CLI 主流程。

## 15. 当前限制与后续方向

当前限制：

- HTML URL 正文抽取未实现。
- Word/PPT 文档解析未实现。
- 多跳 QA 尚未接入扩写和 think 的一体化 CLI 参数。
- Depth / Width QA 尚未接入主 CLI。
- 去重过滤尚未使用 embedding，仅使用轻量文本相似度。
- LLM 输出质量依赖本地模型能力和 prompt 遵循程度。

后续可扩展方向：

- 增加 Markdown 清洗与表格结构化解析。
- 增加批量目录级 Markdown QA 生成入口。
- 将多跳 QA 接入扩写与 think 增强。
- 将 Depth / Width QA 暴露为独立 CLI。
- 增加 ShareGPT、ChatML 等导出格式。
- 增加 embedding-based 去重、多样性采样和主题覆盖控制。
- 增加断点续跑和 LLM 调用缓存。
