# DataLightFlow 配置文件说明

本文档说明 `configs/datalight.example.yaml` 的字段含义、优先级和使用方式。

## 1. 配置目标

配置文件 v1 只覆盖最常用、最稳定的项目级设置：

1. 本地 MinerU 命令路径。
2. 默认输出根目录。
3. LLM 连接参数。
4. QA 构建主题 `topic`。
5. 各阶段可修改的 system prompt。

其它运行参数仍由 CLI 控制，例如：

- chunk 大小
- 最低评分阈值
- 是否启用扩写
- 是否启用 think

这样可以避免配置文件过重，也能保持命令行调试足够灵活。

## 2. 配置文件路径

推荐复制模板：

```bash
cp configs/datalight.example.yaml configs/datalight.yaml
```

运行时通过 `--config` 指定：

```bash
datalight pipeline markdown-qa \
  --config configs/datalight.yaml \
  --markdown demo.md \
  --lmstudio
```

## 3. 优先级

配置优先级为：

```text
CLI 参数 > 配置文件 > 环境变量或代码默认值
```

例如：

- 如果 CLI 传了 `--output-dir`，则使用 CLI 输出目录。
- 如果 CLI 没传 `--output-dir`，则使用 `output.root` 推导出的默认目录。
- 如果 `mineru.executable` 为空，则回退到 `MINERU_EXECUTABLE` 或 `mineru`。

## 4. 完整示例

```yaml
mineru:
  executable: /Users/miller/miniconda3/envs/mineru/bin/mineru

output:
  root: .tmp/datalight_outputs

llm:
  provider: lmstudio
  base_url: http://127.0.0.1:1234/v1
  model: gemma-4-31b-it
  timeout_sec: 180
  temperature: 0.2

qa:
  topic: OpenClaw 的架构、部署、渠道接入、Skills、安全与成本

prompts:
  singlehop_system: |
    你是一个专业的中文 QA 数据构造专家。
    你只围绕以下 Topic 构建高质量问答对：
    {topic}

    如果上下文与 Topic 无关，请返回空的 Q 和 A。

  evaluator_system: |
    你是一个 QA 数据质量评估专家。
    你只评估与以下 Topic 相关的问答对：
    {topic}

  multihop_system: |
    你是一个专业的多跳 QA 构造专家。
    你只围绕以下 Topic 生成需要连接 2-3 个事实的问题：
    {topic}

  expansion_system: |
    你是一个 QA 扩写专家。
    你只扩写与以下 Topic 相关的 QA：
    {topic}

  thinking_system: |
    你负责为 QA 添加简洁 think 字段，并根据 think 重构答案。
    think 和答案都必须围绕以下 Topic：
    {topic}
```

## 5. 字段说明

### `mineru.executable`

本地 MinerU CLI 路径。

```yaml
mineru:
  executable: /Users/miller/miniconda3/envs/mineru/bin/mineru
```

等价于指定本地 MinerU 命令。若不配置，则使用：

1. 环境变量 `MINERU_EXECUTABLE`
2. `PATH` 中的 `mineru`

### `output.root`

默认输出根目录。

```yaml
output:
  root: .tmp/datalight_outputs
```

当 CLI 没传输出路径时，各命令会使用：

```text
output.root/ingest
output.root/qa
output.root/multihop
output.root/expansion/qa_expanded.jsonl
output.root/think/qa_with_think.jsonl
```

对应关系：

| 命令 | 默认输出 |
| --- | --- |
| `ingest directory` | `output.root/ingest` |
| `ingest url` | `output.root/ingest` |
| `pipeline markdown-qa` | `output.root/qa` |
| `pipeline markdown-multihop-qa` | `output.root/multihop` |
| `pipeline expand-qa` | `output.root/expansion/qa_expanded.jsonl` |
| `pipeline add-think` | `output.root/think/qa_with_think.jsonl` |

### `qa.topic`

QA 构建主题。

```yaml
qa:
  topic: OpenClaw 的架构、部署、渠道接入、Skills、安全与成本
```

该字段会替换 system prompt 中的 `{topic}`。

如果为空：

```yaml
qa:
  topic: ""
```

则 `{topic}` 会被替换为空字符串。此时建议同步调整 system prompt，避免出现语义不完整的提示。

### `prompts.*_system`

各阶段的 system prompt。

当前支持：

```yaml
prompts:
  singlehop_system: ...
  evaluator_system: ...
  multihop_system: ...
  expansion_system: ...
  thinking_system: ...
```

各字段含义：

| 字段 | 作用阶段 |
| --- | --- |
| `singlehop_system` | 单跳 QA 生成 |
| `evaluator_system` | 四维 QA 评估 |
| `multihop_system` | 多跳 QA 生成 |
| `expansion_system` | QA 扩写 |
| `thinking_system` | think 字段增强 |

每段 system prompt 都可以显式包含：

```text
{topic}
```

运行时会替换为 `qa.topic`。

### `llm`

LLM 连接配置。

```yaml
llm:
  provider: lmstudio
  base_url: http://127.0.0.1:1234/v1
  model: gemma-4-31b-it
  timeout_sec: 180
  temperature: 0.2
```

字段说明：

| 字段 | 含义 |
| --- | --- |
| `provider` | LLM 提供方。首版支持 `lmstudio` 和 `openai-compatible`。 |
| `base_url` | OpenAI-compatible API base URL。 |
| `model` | 模型名。 |
| `timeout_sec` | 单次请求超时时间，单位秒。 |
| `temperature` | 采样温度。 |

配置了 `llm.provider` 后，LLM 相关命令可以不再显式传 `--lmstudio`：

```bash
datalight pipeline markdown-qa \
  --config configs/datalight.yaml \
  --markdown ./demo.md
```

仍然可以用 CLI 覆盖配置文件：

```bash
datalight pipeline markdown-qa \
  --config configs/datalight.yaml \
  --markdown ./demo.md \
  --llm-model qwen3-32b \
  --llm-timeout 300
```

优先级：

```text
--responses-file > CLI LLM 参数 > config.llm > 代码默认值
```

注意：

- `--responses-file` 用于确定性测试，会优先使用静态响应，不调用真实 LLM。
- `--lmstudio` 仍然可用；当配置中没有 `llm.provider` 时，可以用它启用默认 LM Studio 配置。
- 如果既没有 `--responses-file`，也没有 `--lmstudio`，且配置文件中没有 `llm.provider`，命令会报错。

## 6. 为什么只配置 system prompt

当前版本只开放 system prompt，不开放 user prompt 模板。

原因是 user prompt 中包含严格输出协议，例如：

- 单跳 QA 必须返回 `Q:` / `A:`
- 多跳 QA 必须返回 JSON
- 扩写必须返回 `expanded_question` / `expanded_answer`
- think 必须返回 `think` / `answer`

如果用户直接修改 user prompt，很容易破坏解析逻辑。因此 v1 只允许修改角色、主题和约束类 system prompt。

## 7. 使用示例

### 文档摄取

```bash
datalight ingest directory ./pdfs \
  --config configs/datalight.yaml
```

未传输出目录时，默认写入：

```text
output.root/ingest
```

如果显式传输出目录：

```bash
datalight ingest directory ./pdfs ./custom_ingest \
  --config configs/datalight.yaml
```

则使用 `./custom_ingest`。

### 单跳 QA

```bash
datalight pipeline markdown-qa \
  --config configs/datalight.yaml \
  --markdown ./demo.md \
  --lmstudio
```

默认输出：

```text
output.root/qa
```

### 多跳 QA

```bash
datalight pipeline markdown-multihop-qa \
  --config configs/datalight.yaml \
  --markdown ./demo.md \
  --lmstudio
```

默认输出：

```text
output.root/multihop
```

### 独立扩写

```bash
datalight pipeline expand-qa \
  --config configs/datalight.yaml \
  --input ./qa_scored.jsonl \
  --lmstudio
```

默认输出：

```text
output.root/expansion/qa_expanded.jsonl
```

### 独立 think 增强

```bash
datalight pipeline add-think \
  --config configs/datalight.yaml \
  --input ./qa_expanded.jsonl \
  --lmstudio
```

默认输出：

```text
output.root/think/qa_with_think.jsonl
```

## 8. 常见问题

### system prompt 里必须写 `{topic}` 吗？

不是必须，但推荐写。只有写了 `{topic}`，配置中的 `qa.topic` 才能进入该阶段 prompt。

### 可以配置 chunk 大小和 min-score 吗？

v1 暂不把 chunk、min-score 这类运行调参放进配置文件。继续通过 CLI 控制：

```bash
--chunk-words 512
--min-score 3
```

LLM 连接参数已经支持放进配置文件，也仍然可以被 CLI 覆盖。

### 如果上下文和 topic 无关会怎样？

当前主要依赖 system prompt 约束 LLM：

- 单跳 prompt 建议返回空 Q/A，现有解析会自动跳过。
- 多跳 prompt 建议返回空 JSON。
- 扩写和 think 阶段会尽量保持原记录，并在解析失败时标记失败。

后续可以进一步增加显式 `skip` 协议。
