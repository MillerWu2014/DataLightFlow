# DataLight 配置文件说明

本文档说明 `configs/datalight.yaml` 的字段含义、优先级与使用方式。内容基于 `src/datalight/config.py` 当前实现。

## 1. 配置目标

配置文件覆盖**项目级、相对稳定**的设置：

| 区块 | 作用 |
|------|------|
| `mineru` | 本地 MinerU CLI 路径 |
| `output` | 默认输出根目录 |
| `llm` | LLM 连接参数 |
| `taxonomy` | Taxonomy 生成器所需的分类体系与领域主题 |

**不在配置文件中**（由 API 参数控制）：

- 切块大小（`chunk_words`、`overlap_words`）
- 评分阈值（`min_score`）
- 是否扩写 / think（`expand_qa`、`add_think`）
- 生成器选择（`generator`: `default` / `atomic` / `taxonomy`）
- 每 chunk 问题数（`question_num`）

**已从配置移除**（现位于代码中）：

- `qa.topic`、`prompts.*_system` — 不再解析；prompt 模板见 `src/datalight/pipeline/prompts/`。  
  领域主题现由 **`taxonomy.topic`** 注入 taxonomy 三阶段 prompt。

---

## 2. 配置文件路径

项目默认配置：

```text
configs/datalight.yaml
```

加载方式：

```python
from pathlib import Path
from datalight.service import DatalightService

service = DatalightService(config=Path("configs/datalight.yaml"))
```

或单次调用传入：

```python
from datalight.service import pipeline_markdown_qa

pipeline_markdown_qa(
    markdown=[Path("doc.md")],
    config=Path("configs/datalight.yaml"),
    lmstudio=True,
)
```

未传 `config` 时，使用 `DatalightConfig()` 空默认（无 `output.root`、无 LLM、空 taxonomy）。

> 当前包内**无 Typer CLI**；下文「API 参数」指 `service.py` 函数关键字参数。

---

## 3. 优先级

```text
API 显式参数 > 配置文件 > 代码默认值 / 环境变量
```

示例：

| 场景 | 结果 |
|------|------|
| 传入 `output_dir=Path("out")` | 使用 `out`（单跳实际写入 `out / generator`） |
| 未传 `output_dir`，配置了 `output.root` | 使用 `{root}/qa` 等派生路径 |
| `mineru.executable` 为空 | 回退 `MINERU_EXECUTABLE` → `PATH` 中的 `mineru` |
| 传入 `llm_model="..."` | 覆盖 `llm.model` |
| 传入 `responses_file` | 使用静态响应，不调用在线 LLM |

LLM 客户端解析（`_build_qa_llm_client`）：

1. `responses_file` → `StaticLLMClient`（响应按 `\n---\n` 分段）
2. 否则需满足：`lmstudio=True` **或** 配置中 `llm.provider` 非空
3. 在线模式必须配置 `llm.base_url`
4. 不可同时指定 `responses_file` 与 `lmstudio=True`

## 4. 完整示例

与仓库内 `configs/datalight.yaml` 结构一致（节选）：

```yaml
mineru:
  executable: /Users/miller/miniconda3/envs/mineru/bin/mineru

output:
  root: .output

llm:
  provider: lmstudio
  base_url: http://127.0.0.1:1234/v1
  model: qwen/qwen3.6-35b-a3b
  timeout_sec: 180
  temperature: 0.5

taxonomy:
  topic: 民航领域
  task_type:
    fact_qa: 领域事实精确问答，适合定义、缩写、数字、阈值、岗位职责等短问答
    rule_explain: 规则解释，说明条款含义、适用条件、例外和边界
    procedure: 流程执行，说明步骤、阶段、检查项或审批链路
    scenario: 情景处置，针对异常、冲突、边界条件给出判断依据和建议
    compare: 对比权衡，比较标准、方案、条件或优先级差异
    instruction_following: 领域内指令遵循治理样本
    refusal_uncertainty: 领域不确定性处理，要求说明依据不足、拒绝编造
    abbreviation_definition: 民航缩略语/术语释义
  level1_name:
    规则约束类:
      - level2_name: 阈值与边界
        focus: 时间、数量、距离、容量、限制条件等明确阈值。
        prompt_hint: 仅在原文明确给出数值或边界条件时生成；禁止估算、补全或推测。
  reasoning_style:
    direct: 直接判断或简短依据
    chain_of_thought: 多步推理，适合复杂规则、流程、情景、对比
    rule_based: 明确依据规则/条款推导
    refusal: 不确定性或拒绝编造样本的边界说明
```

## 5. 字段说明

### 5.1 `mineru.executable`

本地 MinerU CLI 可执行文件路径。

```yaml
mineru:
  executable: /path/to/mineru
```

解析顺序：

1. 配置文件 `mineru.executable`
2. 环境变量 `MINERU_EXECUTABLE`
3. `PATH` 中的 `mineru`

### 5.2 `output.root`

默认输出根目录。未在 API 中指定 `output_dir` / `output_path` 时，各流水线回退到此根目录下的子路径。

```yaml
output:
  root: .output
```

| 派生路径（`OutputSettings`） | 用途 |
|------------------------------|------|
| `{root}/ingest` | 目录 / URL 摄取 |
| `{root}/qa` | 单跳 QA 默认根目录（实际还会加 `/generator` 子目录） |
| `{root}/multihop` | 多跳 QA |
| `{root}/expansion/qa_expanded.jsonl` | 独立扩写默认输出文件 |
| `{root}/think/qa_with_think.jsonl` | 独立 think 默认输出文件 |

**单跳特殊约定**：`pipeline_markdown_qa` 写入 **`output_dir / generator`**，例如：

```text
.output/qa/taxonomy/chunks.jsonl
.output/qa/default/qa_export.jsonl
```

同一 `output_dir` 下跑不同 `generator` 不会互相覆盖。

### 5.3 `llm`

OpenAI-compatible Chat Completions 连接配置。

```yaml
llm:
  provider: lmstudio          # 或 openai-compatible
  base_url: http://127.0.0.1:1234/v1
  model: qwen/qwen3.6-35b-a3b
  timeout_sec: 180
  temperature: 0.5
```

| 字段 | 含义 |
|------|------|
| `provider` | 非空即视为 LLM 已配置（`LLMSettings.is_configured()`） |
| `base_url` | API 根 URL，在线调用**必填** |
| `model` | 模型名；API 可传 `llm_model` 覆盖 |
| `timeout_sec` | 单次请求超时（秒）；API 可传 `llm_timeout` 覆盖 |
| `temperature` | 采样温度；未配置时代码默认 `0.2` |

配置了 `llm.provider` 后，调用时可不传 `lmstudio=True`（二者满足其一即可启用在线 LLM）。

代码默认模型名（配置与 API 均未指定时）：`gemma-4-31b-it`。

### 5.4 `taxonomy`

供 `pipeline_markdown_qa(..., generator="taxonomy")` 使用。须满足 `TaxonomySettings.is_complete()`，否则报错。

#### `taxonomy.topic`

领域主题字符串，注入 taxonomy 打标签 / 出题 / 作答 prompt（`pipeline/prompts/taxonomy.py`）。

```yaml
taxonomy:
  topic: 民航领域
```

- 空或仅空白时，回退默认值 **`民航领域`**（`DEFAULT_TAXONOMY_TOPIC`）
- 仅影响 **taxonomy** 生成器；default / atomic / multihop 不使用此字段

#### `taxonomy.task_type`

任务类型枚举：键为程序内合法 `task_type`，值为中文描述（写入 catalog 与 prompt）。

```yaml
task_type:
  fact_qa: 领域事实精确问答，...
  rule_explain: 规则解释，...
```

打标签时 LLM 必须从该 map 的键中选择。

#### `taxonomy.level1_name`

一级分类 → 二级分类列表。YAML 结构：

```yaml
level1_name:
  规则约束类:
    - level2_name: 阈值与边界
      focus: 该二级分类的关注点（必填）
      prompt_hint: 出题倾向提示（必填）
```

解析为 `TaxonomyCategory` 列表；`(level1_name, level2_name)` 必须成对出现在 catalog 中，LLM 打标不得超出此组合。

#### `taxonomy.reasoning_style`

推理风格枚举，键为合法 `reasoning_style`，值为描述。

```yaml
reasoning_style:
  direct: 直接判断或简短依据
  chain_of_thought: 多步推理，...
```

#### 完整性检查 `is_complete()`

以下条件**全部**满足才视为 complete：

- `task_type`、`reasoning_style` 非空
- `categories` 非空
- 每个 category 的 `level1_name`、`level2_name`、`focus`、`prompt_hint` 均非空

## 6. Prompt 与配置的关系

| 内容 | 位置 |
|------|------|
| Taxonomy 领域主题 | `taxonomy.topic` → `prompts/taxonomy.py` |
| Taxonomy 分类 catalog | `taxonomy.*` → `build_taxonomy_catalog()` |
| Text2QA / Atomic / Multihop / Agentic 模板 | `src/datalight/pipeline/prompts/*.py`（**不可通过 YAML 修改**） |
| 扩写 / think user prompt | `generation/expansion.py`、`generation/thinking.py` 内联 |
| 各阶段 system 后缀（如 JSON 格式约束） | Operator 内常量或模板，当前传 `system_prompt=""` |

修改生成行为时：

- **Taxonomy 分类与领域**：改 `configs/datalight.yaml` 的 `taxonomy` 段
- **Prompt 文案与输出协议**：改 Python 源码中的 prompt 模板（需保持 JSON / Q:A 解析兼容）

## 7. 使用示例（Python API）

### 7.1 文档摄取

```python
service.ingest_directory(
    input_dir=Path("data/files"),
    output_dir=Path("data/markdown"),  # 省略则用 output.root/ingest
    config=Path("configs/datalight.yaml"),
    backend="vlm-auto-engine",
)
```

### 7.2 单跳 QA（taxonomy）

```python
result = service.pipeline_markdown_qa(
    markdown=[Path("data/markdown/doc.md")],
    output_dir=Path("outputs/singlehop/doc"),
    config=Path("configs/datalight.yaml"),
    generator="taxonomy",
    chunk_words=2048,
    overlap_words=128,
    question_num=5,
    min_score=3.0,
    expand_qa=True,
    add_think=True,
    lmstudio=True,
)
# 产物：outputs/singlehop/doc/taxonomy/
print(result.export_path)
```

### 7.3 单跳 QA（default / atomic）

```python
# default：两阶段 Text2QA
service.pipeline_markdown_qa(
    markdown=[Path("doc.md")],
    generator="default",
    question_num=3,
    lmstudio=True,
    config=Path("configs/datalight.yaml"),
)

# atomic：内置 recall + golden-doc 验证，跳过四维评估
service.pipeline_markdown_qa(
    markdown=[Path("doc.md")],
    generator="atomic",
    question_num=3,
    atomic_max_per_task=10,
    lmstudio=True,
    config=Path("configs/datalight.yaml"),
)
```

### 7.4 多跳 QA

```python
mh = service.pipeline_markdown_multihop_qa(
    markdown=[Path("doc.md")],
    output_dir=Path("outputs/multihop"),
    config=Path("configs/datalight.yaml"),
    chunk_words=512,
    num_q=5,
    lmstudio=True,
)
```

### 7.5 独立扩写 / think

```python
exp = service.pipeline_expand_qa(
    input_path=mh.generated_path,
    config=Path("configs/datalight.yaml"),
    mode="detail",
    lmstudio=True,
)

think = service.pipeline_add_think(
    input_path=exp.output_path,
    config=Path("configs/datalight.yaml"),
    lmstudio=True,
)
```

未传 `output_path` 时，分别默认 `output.expansion_path()`、`output.think_path()`。

### 7.6 静态响应测试

```python
service.pipeline_markdown_qa(
    markdown=[Path("doc.md")],
    responses_file=Path("tests/fixtures/responses.txt"),
    generator="default",
)
```

`responses.txt` 中多条响应用单独一行 `---` 分隔。

## 8. 配置加载 API

除 `DatalightService` 外，可直接使用：

```python
from pathlib import Path
from datalight.config import DatalightConfig, load_taxonomy

cfg = DatalightConfig.from_file(Path("configs/datalight.yaml"))
taxonomy = cfg.taxonomy_data()       # TaxonomySettings
only_taxonomy = load_taxonomy(Path("configs/datalight.yaml"))
```

## 9. 常见问题

### `generator="taxonomy"` 报错「requires a complete taxonomy config」

检查 `taxonomy` 段是否缺少 `task_type`、`reasoning_style`、`level1_name` 或某个 category 的 `focus` / `prompt_hint` 为空。

### 修改 `topic` 后 default 生成器没有变化？

`taxonomy.topic` 仅作用于 taxonomy 流水线。default / atomic 使用 `pipeline/prompts/text2qa.py` 等模板，与 `topic` 无关。

### 能否在 YAML 里配置 system prompt？

当前版本**不支持**。历史字段 `prompts.*_system`、`qa.topic` 已被移除，解析代码不再读取。

### 能否在 YAML 里配置 chunk_words、min_score？

**不能**。通过 API 参数传入，例如 `chunk_words=512`、`min_score=3.0`。

### 未配置 `output.root` 且未传 `output_dir` 会怎样？

抛出 `ValueError`：`Provide output_dir or set output.root in config`。

### 多 generator 输出会互相覆盖吗？

不会。单跳产物在 `{output_dir}/{generator}/` 下分目录存放。

## 10. 与旧版文档的差异

| 旧版 | 当前 |
|------|------|
| `configs/datalight.example.yaml` | 使用 `configs/datalight.yaml` |
| `qa.topic` + `{topic}` 占位符 | `taxonomy.topic`，仅 taxonomy prompt |
| `prompts.*_system` | 已删除；prompt 在代码中 |
| CLI `datalight pipeline ...` | `DatalightService` / `service.py` 函数 |
| 单跳输出 `{root}/qa/` 根目录 | 实际为 `{root}/qa/{generator}/`（或自定义 `output_dir/generator`） |

更多架构细节见 [技术实现方案.md](./技术实现方案.md)。
